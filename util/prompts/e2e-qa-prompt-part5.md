## STEP 6 — Heal Failing Tests

### 6.1 Purpose

After the initial full suite run, some tests may fail due to:
- Selector changes (DOM restructuring or attribute drift)
- Timing issues (elements not yet rendered when the assertion fires)
- Test data mismatches (application under test has changed expected values)
- Logic errors in the generated spec code
- Environment or configuration issues

The **`playwright-test-healer`** agent must diagnose each failure, apply the minimal fix, re-run the test, and log every change.

### 6.2 Pre-conditions

1. `playwright.config.ts` is configured with:
   - `screenshot: 'on'`
   - `video: 'on'`
   - `trace: 'on'`
   - `outputDir: 'reports/test-results'`
   - `reporter: [['html'], ['json', { outputFile: 'reports/test-results/initial-run-report.json' }]]`
2. All 19 spec files exist under `tests/checkout/`.
3. The happy-path spec passes (verified at end of Part 4).

### 6.3 Step 1 — Run the Full Suite

1. Run the full Playwright test suite targeting `tests/checkout/`.
2. Capture the complete stdout output.
3. Save the Playwright JSON report to `reports/test-results/initial-run-report.json`.
4. Note: The Playwright HTML report is written to `playwright-report/` automatically.
5. Record:
   - Total tests run
   - Pass count
   - Fail count
   - Flaky count (if any)
   - Skipped count

### 6.4 Step 2 — Triage Every Failure

For each failing test, create a triage entry in `reports/test-results/failure-triage.md` using the template below.

```
FAILURE-<SEQ>
───────────────────────────────────────────────
Test ID          : <TC-AC_NUMBER-SEQ from test plan>
Test File        : tests/checkout/<filename>.spec.ts
Test Title       : <exact test title from spec file>
Failure Type     : <Selector | Timing | Data | Logic | Environment | Unknown>
Root Cause       : <one-paragraph explanation of why the test failed>
Error Message    : <exact error thrown by Playwright>
Evidence File    : <path to screenshot or trace in reports/test-results/>
Fix Applied      : <description of the change made to fix the test>
Lines Changed    : <file:line ranges modified>
───────────────────────────────────────────────
```

#### Root-Cause Categories (RCA)

| Category | Description | Typical Fix |
|---|---|---|
| A — Selector | Element not found because `data-test` value, class name, or DOM structure changed | Update locator in page object |
| B — Timing | Element not yet visible or interactable when assertion fires | Replace with auto-waiting `expect(locator).toBeVisible()` or `waitForURL` |
| C — Data | Application returns a value different from what is in `checkout-test-data.json` | Update `checkout-test-data.json` to match actual application output |
| D — Logic | Test performs actions in wrong order, or makes wrong assertion | Fix action sequence or assertion in spec file |
| E — Config | Playwright config missing a required setting | Fix `playwright.config.ts` |
| F — Environment | App is temporarily unreachable, network error, or login session expired | Retry; if persistent, document as a known environment issue |

### 6.5 Step 3 — Heal and Verify

For each triage entry:

1. Apply the **minimal fix** — change only what is necessary to make the test pass.
2. Re-run that single test in isolation.
3. If it passes, proceed.
4. If it fails again, attempt a second fix using a different approach.
5. If it fails three times, mark it as `BLOCKED` in the triage doc and move on.
6. Log every change in `reports/test-results/healing-log.md` using this format:

```
HEAL-<SEQ>
───────────────────────────────────────────────
Triage Ref       : FAILURE-<SEQ>
Test ID          : <TC-AC_NUMBER-SEQ>
File Modified    : <filepath>
Change Type      : <Selector update | Wait strategy | Data correction | Logic fix | Config fix>
Before           :
  <original code block>
After            :
  <new code block>
Reason           : <why this specific change fixes the root cause>
Verified Pass    : <Yes | No | BLOCKED>
───────────────────────────────────────────────
```

### 6.6 Step 4 — Post-Healing Verification

After all individual test fixes are applied:

1. Re-run the **full suite** again (`tests/checkout/`).
2. Save the Playwright JSON report to `reports/test-results/post-healing-run-report.json`.
   (Update `playwright.config.ts` reporter output path temporarily, or use `--reporter=json` with `--output` CLI flag.)
3. Record:
   - Total tests run
   - Pass count (must be ≥ initial pass count)
   - Fail count (must be ≤ initial fail count)
   - Any new failures introduced by healing changes (regression check)

### 6.7 Healing Rules

The agent MUST follow these rules when applying fixes. Violations invalidate the report.

1. **Minimum change principle** — fix only the line(s) directly causing the failure. Do not refactor, rename, or restructure passing code.
2. **No new `waitForTimeout`** — all timing fixes must use Playwright's built-in auto-waiting mechanisms.
3. **One change per heal cycle** — apply one fix, re-run, then assess. Do not batch multiple speculative changes.
4. **Selector changes go in the page object** — never hard-code a selector inside a spec file.
5. **Data corrections go in the JSON** — never hard-code expected values inside a spec file.
6. **Document every change** — no undocumented modifications.
7. **Do not skip or `test.skip()` tests** — if a test cannot be healed, mark it `BLOCKED` but do not suppress it.

### 6.8 Acceptance Criteria for Step 6

Step 6 is complete when:

- [ ] `reports/test-results/failure-triage.md` exists with one entry per failure
- [ ] `reports/test-results/healing-log.md` exists with one `HEAL-*` entry per change applied
- [ ] `reports/test-results/post-healing-run-report.json` exists
- [ ] Post-healing pass count is ≥ 90% of total tests (or all tests pass)
- [ ] No new failures were introduced by healing changes
- [ ] No test is marked `test.skip()` or silently ignored
